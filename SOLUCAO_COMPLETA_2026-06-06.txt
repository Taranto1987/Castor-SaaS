SOLUCAO P1 — PASSO 1: CONFIRMAR (READ-ONLY, seguro) via agente Railway
--------------------------------------------------------------------------
Cole no agente Railway / psql (somente leitura):

  -- (A) a coluna existe mesmo? (esperado: vendido_em AUSENTE)
  SELECT column_name, data_type
  FROM information_schema.columns
  WHERE table_name = 'orcamentos'
  ORDER BY ordinal_position;

  -- (B) contagens reais (validacao geral de dados)
  SELECT 'leads' t, count(*) FROM leads
  UNION ALL SELECT 'sales_opportunities', count(*) FROM sales_opportunities
  UNION ALL SELECT 'orcamentos', count(*) FROM orcamentos
  UNION ALL SELECT 'entregas', count(*) FROM entregas
  UNION ALL SELECT 'sessions', count(*) FROM sessions
  UNION ALL SELECT 'lojas', count(*) FROM lojas;

  -- (C) consistencia multi-tenant
  SELECT loja_id, count(*) FROM leads GROUP BY loja_id;
  SELECT loja_id, count(*) FROM orcamentos GROUP BY loja_id;

SOLUCAO P1 — PASSO 2: CORRIGIR (apos confirmar) via agente Railway
--------------------------------------------------------------------------
Comando unico, aditivo, idempotente, reversivel:

  ALTER TABLE orcamentos ADD COLUMN IF NOT EXISTS vendido_em timestamp;

POR QUE E SEGURO (RISCO BAIXISSIMO):
  - E exatamente o que o schema Drizzle declara: timestamp("vendido_em"),
    nullable, sem default.
  - Nullable -> nao quebra linhas existentes; o codigo ja faz
    COALESCE(vendido_em, criado_em), entao registros antigos seguem
    funcionando SEM backfill.
  - IF NOT EXISTS -> idempotente; nao erra se ja existir.
  - Nao dropa, nao recria, nao altera tipo -> zero risco de perda de dados.

VERIFICACAO POS-FIX (read-only):
  SELECT column_name FROM information_schema.columns
  WHERE table_name='orcamentos' AND column_name='vendido_em';   -- deve retornar 1 linha

  Depois: abrir Financeiro/DRE no app e confirmar que nao ha mais 500.
  Conferir nos logs que scheduler_cycle_error parou de aparecer.

ROLLBACK (se necessario, improvavel):
  ALTER TABLE orcamentos DROP COLUMN IF EXISTS vendido_em;

  faltantes mas NAO adiciona colunas novas em tabelas existentes. Por isso
  qualquer coluna adicionada ao schema Drizzle depois da tabela ja existir
  (como vendido_em) nunca chega ao banco -> P1 se repete a cada coluna nova.

SOLUCAO (escolher 1):
  OPCAO 1 (minima, mantem o fluxo atual) — fazer o reconcile detectar colunas
    do schema ausentes no banco e aplicar:
      ALTER TABLE <tabela> ADD COLUMN IF NOT EXISTS <col> <tipo>;
    Comparar colunas do schema Drizzle vs information_schema.columns e gerar os
    ALTERs idempotentes. Log: "[reconcile] colunas adicionadas=N".

  OPCAO 2 (trocar o motor) — usar `drizzle-kit push` real no pre-deploy, que ja
    faz diff de colunas automaticamente, no lugar do reconcile.mjs custom.
    Avaliar antes se o drizzle-kit push nao tenta operacoes destrutivas no
    schema atual (rodar com --verbose em staging/local primeiro).

RECOMENDACAO: Opcao 1 (menor risco, mantem o comportamento conhecido).
